CI/CD Pipelines with Tekton

Automate Build, Test & Deploy on GKE

What is CI/CD?

The backbone of modern software delivery

CI/CD stands for Continuous Integration and Continuous Delivery/Deployment. It is a set of practices that automate the process of integrating code changes, testing them, and delivering them to production — quickly, reliably, and repeatedly.

⚙
Continuous Integration

Developers merge code to a shared branch frequently. Each merge triggers an automated build and test cycle, catching bugs early before they reach production.

🚀
Continuous Delivery / Deployment

Every code change that passes tests is automatically packaged and either staged for release (Delivery) or deployed directly to production (Deployment).

Why it matters: Without CI/CD, releases are manual, error-prone, and infrequent. With CI/CD, teams ship multiple times per day with confidence and safety.

The Problem Without CI/CD

Manual deployments are slow, risky, and stressful

Without CI/CD

  • Developer builds JAR locally: mvn package
  • Manually copies JAR to server via SCP
  • SSHs into server, stops old process, starts new one
  • Forgets to run tests — bug reaches production
  • Rollback means finding the old JAR somewhere
  • "It works on my machine" syndrome
  • Release day = stressful all-hands event

With CI/CD

  • Developer pushes code to Git
  • Pipeline automatically builds, tests, packages
  • Docker image pushed to registry automatically
  • Deployment happens with zero downtime
  • Rollback is one click or automatic
  • Consistent environments every time
  • Release is a non-event — happens 10x daily
Real-world impact: A 2023 DORA report found that elite teams deploy 973x more frequently than low performers, with 6,570x faster lead time from commit to production.

CI vs CD vs CD

Three related but distinct practices

🔄
Continuous Integration (CI)

Focus: Merging & Testing

Developers integrate code into a shared repo multiple times a day. Each integration triggers an automated build and test suite. Goal: catch integration bugs early.

Output: A verified, tested build artifact

📦
Continuous Delivery (CD)

Focus: Release Readiness

Extends CI by automatically deploying all code changes to a staging environment after the build and test stages. Production deploy requires a manual approval click.

Output: A release-ready artifact in staging

🚀
Continuous Deployment (CD)

Focus: Full Automation

Goes one step further — every change that passes all tests is automatically deployed to production with zero human intervention. Requires mature testing and monitoring.

Output: Code in production within minutes

Pipeline Stages In Detail

What happens at each stage for our UPI Transfer Service

1
Source (Clone)
Clone the Git repository. Checkout the specific branch or commit that triggered the pipeline. This gives us a clean, reproducible starting point.
2
Build (Compile)
Run mvn compile to compile the Java source code. Resolves all Maven dependencies. Catches syntax errors and missing imports immediately.
3
Test (Unit + Integration)
Run mvn test to execute all JUnit tests. Code coverage reports are generated. Pipeline fails if any test fails — no exceptions.
4
Package (Docker Image)
Use Kaniko to build a Docker image inside Kubernetes (no Docker daemon needed). The Dockerfile uses a multi-stage build for a minimal production image.
5
Push (Artifact Registry)
Tag the image with the Git commit SHA and push to Google Artifact Registry. Immutable tags ensure traceability from image back to exact source code.
6
Deploy (Cloud Run)
Deploy the new image to Cloud Run using gcloud run deploy. Zero-downtime rolling update. Automatic HTTPS endpoint provisioned.

Cloud-Native CI/CD

Why Kubernetes-native pipelines are the future

Traditional CI/CD tools (Jenkins, Bamboo) were designed for a VM-based world. They use long-lived servers, persistent agents, and plugin-heavy architectures. Cloud-native CI/CD flips this model by treating pipelines as Kubernetes workloads.

💻
Traditional CI/CD

• Runs on dedicated VMs or servers
• Agents must be provisioned and maintained
• Scaling means adding more VMs
• Configuration stored in UI or Groovy scripts
• Plugins can conflict and break upgrades
• Not portable across cloud providers

☵
Cloud-Native CI/CD (Tekton)

• Runs as Pods on Kubernetes
• Each pipeline step is a container
• Auto-scales with the cluster
• Configuration is YAML + CRDs (GitOps-friendly)
• Steps are isolated containers — no conflicts
• Portable: runs on any Kubernetes cluster

Key insight: If your app runs on Kubernetes, your CI/CD pipeline should too. Same infrastructure, same tooling, same security model.

What is Tekton?

A Kubernetes-native, open-source CI/CD framework

Tekton is an open-source framework for creating CI/CD systems. It was originally developed by Google as part of the Knative project, then donated to the Continuous Delivery Foundation (CDF). It installs as a set of Custom Resource Definitions (CRDs) on any Kubernetes cluster.

⚙
Kubernetes-Native

Tekton runs entirely on Kubernetes. Pipelines are Kubernetes resources (CRDs). Each step runs in its own container inside a Pod. kubectl is your CLI.

🔌
Open Source

Part of the CD Foundation (Linux Foundation). Vendor-neutral — no lock-in to any cloud provider. Active community with regular releases and a growing catalog.

🚀
Declarative Pipelines

Define your entire CI/CD pipeline in YAML. Version control it alongside your application code. Apply with kubectl apply -f — GitOps ready.

Think of it this way: Tekton is to CI/CD what Kubernetes is to container orchestration — a standardized, extensible platform that runs anywhere.

Tekton Architecture Overview

How Tekton fits into a Kubernetes cluster

Developer
git push / kubectl apply
↓
Kubernetes API Server
Receives CRD objects
↓
Tekton Controller
Watches for PipelineRun CRDs
↓
Pod Scheduler
Creates Pods for each Task
↓
Task Pods
Each step = a container in the Pod
Key architecture points:
• Tekton installs a controller in the tekton-pipelines namespace
• The controller watches for Pipeline/Task CRDs via the Kubernetes API
• When a PipelineRun is created, the controller creates Pods to execute each Task
• Each Step within a Task becomes a container within that Task's Pod
• Results and logs are stored as Kubernetes annotations and container logs

Tekton Building Blocks

The five core concepts you need to know

▶
Step

The smallest unit of execution. A single container that runs a command. Example: mvn test or docker build. Steps run sequentially within a Task.

📝
Task

A collection of ordered Steps that run in a single Pod. Like a function — it has inputs (params), outputs (results), and a body (steps). Reusable across pipelines.

🔗
Pipeline

An ordered collection of Tasks. Defines the full CI/CD workflow. Tasks can run sequentially or in parallel. Defines how data flows between Tasks via Workspaces.

▶
PipelineRun

An execution instance of a Pipeline. Provides concrete values for parameters and workspaces. Created manually or automatically via Triggers. Each run is immutable and auditable.

📁
Workspace

Shared storage between Tasks in a Pipeline. Backed by PersistentVolumeClaims, ConfigMaps, or Secrets. Used to pass files (like source code or build artifacts) between Tasks.

Analogy: Step = a line of code, Task = a function, Pipeline = a program, PipelineRun = executing the program, Workspace = shared variables.

Tekton Step

The smallest unit of work — a single container execution

A Step is a single container that runs a specific command. Steps run sequentially within a Task, sharing the same Pod (and therefore the same volumes and network namespace). 

# A Step runs inside a Task definition
steps:
  - name: run-tests
    image: maven:3.9-eclipse-temurin-17
    workingDir: $(workspaces.source.path)
    command:
      - /bin/bash
    args:
      - -c
      - |
        mvn test -B -q
        echo "All tests passed!"
📦
image

Any Docker image. The container runtime for this step. Use official images like maven, node, gcr.io/kaniko.

💻
command / args

The command to execute inside the container. Equivalent to Dockerfile CMD. Can be a shell script.

📁
workingDir

The directory where the command runs. Typically set to the workspace path where source code is cloned.

Tekton Task

A reusable collection of steps — like a function

apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: maven-build-test
spec:
  workspaces:
    - name: source
      description: The workspace with the cloned source code
  params:
    - name: GOALS
      type: string
      default: "clean package -DskipTests=false"
  steps:
    - name: maven-build
      image: maven:3.9-eclipse-temurin-17
      workingDir: $(workspaces.source.path)
      command:
        - mvn
      args:
        - $(params.GOALS)
Key points: • Tasks have params (inputs), workspaces (shared storage), and steps (actions)
• Tasks are reusable — define once, use in many pipelines
• Each Task runs as a single Pod — all steps share the same volume mounts

Tekton Pipeline

Orchestrating multiple tasks into a complete workflow

apiVersion: tekton.dev/v1beta1
kind: Pipeline
metadata:
  name: upi-service-pipeline
spec:
  workspaces:
    - name: shared-workspace
  params:
    - name: git-url
    - name: git-revision
      default: main
    - name: image-name
  tasks:
    - name: clone
      taskRef:
        name: git-clone        # from Tekton Hub
      workspaces:
        - name: output
          workspace: shared-workspace
    - name: build-test
      taskRef:
        name: maven-build-test  # our custom Task
      runAfter: ["clone"]
      workspaces:
        - name: source
          workspace: shared-workspace
runAfter: Controls task ordering. Tasks without runAfter run in parallel. Tasks with runAfter wait for the specified task to complete first.

Tekton PipelineRun

Executing a pipeline with concrete values

A PipelineRun is what actually triggers a pipeline. It provides the concrete values for parameters and workspaces. Think of Pipeline as a class and PipelineRun as an instance. 

apiVersion: tekton.dev/v1beta1
kind: PipelineRun
metadata:
  generateName: upi-service-run-
spec:
  pipelineRef:
    name: upi-service-pipeline
  params:
    - name: git-url
      value: https://github.com/ford-team/upi-transfer-service.git
    - name: git-revision
      value: main
    - name: image-name
      value: us-central1-docker.pkg.dev/ford-project/upi-repo/upi-service
  workspaces:
    - name: shared-workspace
      persistentVolumeClaim:
        claimName: tekton-pvc
generateName: Using generateName instead of name lets Kubernetes append a unique suffix, so every run gets a unique identifier (e.g., upi-service-run-7xk2m).

Tekton Workspaces

Sharing data between tasks in a pipeline

Since each Task runs in its own Pod, tasks cannot share files via the local filesystem. Workspaces provide a shared volume that multiple tasks can read from and write to.

git-clone
writes source to workspace
→
maven-build
reads source, writes JAR
→
kaniko-build
reads JAR, builds image
💾
PersistentVolumeClaim

Most common option. A PVC provides persistent storage backed by a disk. Data survives across task pods. Use ReadWriteOnce for sequential tasks.

📄
ConfigMap

For read-only configuration files. Mount a ConfigMap as a workspace to provide config files (like settings.xml) to build tasks.

🔒
Secret

For sensitive data like registry credentials or SSH keys. Mounted as files in the workspace. Kubernetes handles encryption at rest.

# PVC for Tekton workspace (apply before running pipeline)
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: tekton-pvc
spec:
  accessModes: [ReadWriteOnce]
  resources:
    requests:
      storage: 1Gi

Tekton Triggers

Automatically run pipelines on git push events

Tekton Triggers extends Tekton Pipelines by enabling event-driven automation. When a developer pushes code to GitHub, a webhook fires, and Tekton automatically creates a PipelineRun.

GitHub Webhook
POST on git push
↓
EventListener
Receives HTTP webhook
↓
TriggerBinding
Extracts params from payload
↓
TriggerTemplate
Creates PipelineRun with params
↓
PipelineRun
Pipeline executes automatically
Three components:
• EventListener: A Kubernetes Service that listens for incoming webhooks
• TriggerBinding: Maps fields from the webhook JSON payload to pipeline parameters
• TriggerTemplate: A template for creating PipelineRun objects with the extracted parameters

Step 1: Clone

Using the git-clone task from Tekton Hub

The first task in our pipeline clones the UPI Transfer Service repository into a shared workspace. We use the official git-clone task from Tekton Hub — no need to write it ourselves. 

# Install the git-clone task from Tekton catalog
kubectl apply -f \
  https://raw.githubusercontent.com/tektoncd/catalog/main/task/git-clone/0.9/git-clone.yaml

# How it appears in our Pipeline definition
tasks:
  - name: clone-repo
    taskRef:
      name: git-clone
    workspaces:
      - name: output
        workspace: shared-workspace
    params:
      - name: url
        value: $(params.git-url)
      - name: revision
        value: $(params.git-revision)
      - name: deleteExisting
        value: "true"
Why deleteExisting? When reusing a PVC across runs, old files may remain. Setting deleteExisting: true ensures a clean clone every time. The workspace now contains the full source code of our UPI Transfer Service, ready for the next task.

Step 2: Build & Test

Maven build with unit tests for UPI Transfer Service

This custom Task compiles the Spring Boot application and runs all JUnit tests. If any test fails, the pipeline stops immediately — no broken code reaches the Docker image stage. 

apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: maven-build-test
spec:
  workspaces:
    - name: source
    - name: maven-settings
      optional: true
  params:
    - name: GOALS
      default: "clean package -B"
    - name: CONTEXT_DIR
      default: "."
  steps:
    - name: build-and-test
      image: maven:3.9-eclipse-temurin-17
      workingDir: $(workspaces.source.path)/$(params.CONTEXT_DIR)
      command: ["mvn"]
      args: ["$(params.GOALS)"]
      resources:
        requests:
          memory: 512Mi
          cpu: 500m
Note: The -B flag enables Maven batch mode (non-interactive), which is essential for CI environments. The built JAR file is written to target/ inside the workspace, ready for Docker packaging.

Step 3: Build Docker Image

Kaniko — build images without a Docker daemon

On Kubernetes, you cannot run docker build because there is no Docker daemon. Kaniko solves this by building images entirely in userspace inside a container. 

apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: kaniko-build-push
spec:
  workspaces:
    - name: source
    - name: dockerconfig     # registry credentials
  params:
    - name: IMAGE
    - name: DOCKERFILE
      default: ./Dockerfile
    - name: CONTEXT
      default: ./
  steps:
    - name: build-and-push
      image: gcr.io/kaniko-project/executor:latest
      args:
        - --dockerfile=$(params.DOCKERFILE)
        - --context=$(workspaces.source.path)/$(params.CONTEXT)
        - --destination=$(params.IMAGE):$(params.TAG)
        - --cache=true
        - --cache-ttl=24h
✅
Why Kaniko?

No privileged mode needed. No Docker socket mounting. Builds in userspace. Safe for multi-tenant Kubernetes clusters. Supports layer caching for faster builds.

🔒
Registry Auth

Kaniko reads credentials from /kaniko/.docker/config.json. On GKE, use Workload Identity to authenticate to Artifact Registry without managing secrets.

Step 4: Push to Artifact Registry

Store container images securely in GCP

Kaniko pushes the image directly to Google Artifact Registry as part of the build step. The image is tagged with the Git commit SHA for full traceability. 

# Create Artifact Registry repository (one-time setup)
gcloud artifacts repositories create upi-repo \
  --repository-format=docker \
  --location=us-central1 \
  --description="UPI Transfer Service images"

# Image naming convention
# REGION-docker.pkg.dev/PROJECT/REPO/IMAGE:TAG
us-central1-docker.pkg.dev/ford-project/upi-repo/upi-service:abc123f

# Configure Docker auth for Artifact Registry
gcloud auth configure-docker us-central1-docker.pkg.dev
🏷
Tagging Strategy

Tag with Git commit SHA (:abc123f). Every image is traceable to the exact source code. Never use :latest in production.

🔎
Vulnerability Scanning

Artifact Registry has built-in vulnerability scanning. Every pushed image is automatically scanned for CVEs. Block deployments of vulnerable images.

🔒
IAM Access Control

Use IAM to control who can push and pull images. Grant roles/artifactregistry.writer to the Tekton service account, reader to Cloud Run.

Step 5: Deploy to Cloud Run

Zero-downtime deployment with a single gcloud command

apiVersion: tekton.dev/v1beta1
kind: Task
metadata:
  name: deploy-cloud-run
spec:
  params:
    - name: IMAGE
    - name: SERVICE_NAME
      default: upi-transfer-service
    - name: REGION
      default: us-central1
    - name: PROJECT_ID
  steps:
    - name: deploy
      image: gcr.io/google.com/cloudsdktool/cloud-sdk:slim
      command:
        - gcloud
      args:
        - run
        - deploy
        - $(params.SERVICE_NAME)
        - --image=$(params.IMAGE)
        - --region=$(params.REGION)
        - --project=$(params.PROJECT_ID)
        - --platform=managed
        - --port=8080
        - --memory=512Mi
        - --allow-unauthenticated
What happens during deploy:
• Cloud Run pulls the new image from Artifact Registry
• A new revision is created with the updated image
• Traffic is gradually shifted to the new revision (zero downtime)
• The old revision is kept for instant rollback if needed
• HTTPS endpoint is automatically provisioned and updated

Pipeline Security

Service accounts, RBAC, and secrets management

1
Kubernetes Service Account
Create a dedicated ServiceAccount for Tekton pipelines. Bind it to only the roles it needs. Never use the default service account for pipeline workloads.
2
Workload Identity
On GKE, use Workload Identity to map the K8s ServiceAccount to a GCP Service Account. No need to export JSON keys — authentication happens via metadata server automatically.
3
RBAC (Role-Based Access Control)
Define ClusterRoles and RoleBindings to control who can create Pipelines, Tasks, and PipelineRuns. Developers can trigger runs but not modify pipeline definitions.
4
Secrets Management
Store registry credentials, API keys, and database passwords as Kubernetes Secrets. Mount them as Workspaces or environment variables. Never hardcode secrets in YAML.

Pipeline Best Practices

Patterns for efficient, reliable CI/CD pipelines

1
Layer Caching
Use Kaniko's --cache=true to cache Docker layers between builds. Use Maven's .m2 directory as a workspace to cache dependencies. This can reduce build times by 50-70%.
2
Parallel Tasks
Run independent tasks in parallel using Tekton's DAG execution model. For example, run linting and unit tests simultaneously if they don't depend on each other.
3
Fail Fast
Order tasks so that the fastest checks (lint, compile) run first. If compilation fails, don't waste time running expensive integration tests. Use runAfter wisely.
4
Immutable Tags
Tag images with the Git commit SHA, not :latest. This ensures every deployment is traceable and reproducible. You can always tell exactly which code is running in production.
5
Notifications
Use a finally block in your Pipeline to send Slack/email notifications regardless of whether the pipeline succeeded or failed. Keep the team informed.
6
Resource Limits
Set CPU and memory requests and limits on every step. Prevent runaway builds from consuming the entire cluster. Start small and tune based on metrics.

Real-World CI/CD at Scale

How enterprises use CI/CD in production

🏢
Multi-Environment Pipelines

Enterprise pipelines deploy through multiple environments: dev → staging → UAT → production. Each environment has its own config (database URLs, API keys, feature flags). Promotion between environments requires automated tests and sometimes manual approval gates.

🔗
Monorepo Pipelines

Large teams use monorepos with path-based pipeline triggers. A change to /services/upi/ triggers only the UPI pipeline. A change to /services/fraud/ triggers only the fraud pipeline. Shared libraries trigger all dependent pipelines.

🚨
Canary & Blue-Green Deploys

Instead of deploying to all users at once, enterprises use canary releases (send 5% of traffic to the new version, monitor, then gradually increase) or blue-green (run two identical environments and switch traffic instantly).

🔒
Compliance & Audit

Regulated industries (banking, healthcare) require audit trails for every deployment. CI/CD pipelines provide this automatically — every PipelineRun records who triggered it, what code was deployed, and what tests passed. Immutable build history.

Ford context: In automotive and fintech, traceability and compliance are critical. A CI/CD pipeline with Tekton provides auditable, reproducible deployments — exactly what regulators want.

Common Pitfalls

Mistakes to avoid when building CI/CD pipelines

Pitfall 1: Skipping tests to "go faster"
Using -DskipTests in CI defeats the entire purpose. If tests are slow, optimize them (parallelize, use test containers, mock external services) — never skip them.
Pitfall 2: Using :latest tag
The :latest tag is mutable — you cannot tell which version is running. Always tag with the Git commit SHA or semantic version. :latest in production = debugging nightmare.
Pitfall 3: Hardcoding secrets in YAML
Never put passwords, API keys, or credentials directly in pipeline YAML files. Use Kubernetes Secrets, GCP Secret Manager, or external vault solutions (HashiCorp Vault).
Pitfall 4: No resource limits on pipeline tasks
A Maven build without memory limits can OOM-kill other pods on the node. Always set resources.requests and resources.limits on every step.
Pitfall 5: Monolithic pipelines
Putting everything in one giant Task makes it impossible to reuse, debug, or parallelize. Break pipelines into small, focused Tasks. Each Task should do one thing well.
