Memorystore & Redis

In-Memory Caching & Rate Limiting for UPI

The Problem

Why do we need an in-memory store?

⏳
Slow Repeated Queries

Every GET /api/balance/nag@upi hits the database — even if balance hasn't changed

🚨
No Rate Limiting

A user can fire 1000 transfers/sec — no throttle, no protection against abuse

📊
Database Under Load

As users grow, DB connections become the bottleneck — reads are 90% of traffic

💰
Cost

Cloud SQL charges per connection-second — unnecessary reads waste money

Key Insight: Most balance lookups return the same data. Cache it in memory → sub-millisecond response, zero DB load.

What is Redis?

The world's most popular in-memory data store

1
Key-Value Store

Simple: SET key value, GET key. Strings, hashes, lists, sets, sorted sets.

2
In-Memory = Fast

All data lives in RAM. Reads in < 1ms. Compare: database = 5-50ms.

3
TTL (Time-To-Live)

Keys auto-expire. SET balance:nag@upi 10000 EX 300 — gone after 5 minutes.

4
Atomic Operations

INCR is atomic — perfect for counters, rate limiters. No race conditions.

Redis is not a database replacement — it's a speed layer that sits between your app and the database.

Rate Limiting with Redis

Protect your service from abuse using atomic counters

Redis Commands 
# First transfer in window
INCR ratelimit:transfer:nag@upi   → 1
EXPIRE ratelimit:transfer:nag@upi 60

# Subsequent transfers
INCR ratelimit:transfer:nag@upi   → 2
INCR ratelimit:transfer:nag@upi   → 3
...
INCR ratelimit:transfer:nag@upi   → 11
# ❌ BLOCKED — count > 10

# After 60s, key expires → resets to 0
✓
Atomic INCR

No race condition — even with 100 concurrent requests, count is exact

!
Fail Open

If Redis is down, allow transfers. Don't block payments because cache is unavailable.

↺
Auto Reset

TTL = 60s. Key expires naturally. No cleanup jobs needed.

Caching Strategies

Choose the right pattern for your use case

🔎
Cache-Aside (Lazy)

We use this. App checks cache first. On miss, reads DB and populates cache. Simple, flexible.

✍
Write-Through

Every DB write also writes to cache. Always fresh, but adds write latency.

🕑
Write-Behind

Write to cache first, async flush to DB. Fastest writes, risk of data loss.

🗑
Cache Invalidation

We also do this. On transfer, evict sender + receiver cache. Next read repopulates.

The Two Hard Problems: Cache invalidation and naming things. We handle invalidation by evicting on write — simple and correct.

Production Considerations

Things to think about before going live

1
Eviction Policy

allkeys-lru: when memory is full, evict least recently used keys. Best for caching.

2
TTL Strategy

Too short = too many cache misses. Too long = stale data. We use 5 min + evict on write.

3
Fail Open

If Redis is down, the app still works — just slower. Never block business logic on cache.

4
Stampede Protection

When a popular key expires, 1000 requests hit DB simultaneously. Solutions: lock, jitter TTL.

5
Monitoring

Track hit rate, memory usage, evictions, latency. Memorystore integrates with Cloud Monitoring.

6
Serialization

JSON is readable but larger. Binary (Protobuf/Kryo) is faster. We use JSON for clarity.

Deploying on GCP

Cloud Run + VPC Connector + Memorystore

bash 
# 1. Create Memorystore instance
gcloud redis instances create upi-cache \
  --size=1 --region=us-central1 \
  --redis-version=redis_7_0 --tier=BASIC

# 2. Create VPC Connector (bridge for Cloud Run)
gcloud compute networks vpc-access connectors \
  create upi-connector \
  --region=us-central1 --range=10.8.0.0/28

# 3. Get Redis IP
export REDIS_HOST=$(gcloud redis instances \
  describe upi-cache --region=us-central1 \
  --format="value(host)")

# 4. Deploy to Cloud Run with VPC Connector
gcloud run deploy upi-service-redis \
  --image=gcr.io/$PROJECT_ID/upi-service-redis \
  --vpc-connector=upi-connector \
  --set-env-vars="REDIS_HOST=$REDIS_HOST"

Memorystore Monitoring

What to watch in production

95%+
Cache Hit Ratio

Target hit rate. Below 80% means TTL is too short or keys are evicted too fast.

< 1ms
Latency

Redis p99 latency. If it spikes, check memory pressure or network issues.

0
Evictions

Keys evicted due to memory full. If non-zero, increase instance size or reduce TTL.

1
Cloud Console

Memorystore → Redis → upi-cache → Monitoring tab. Memory, clients, commands/sec.

2
Redis CLI

redis-cli -h <ip> INFO stats — keyspace hits/misses, commands processed.

3
Cloud Run Logs

Our app logs CACHE HIT/MISS/EVICT — visible in Cloud Logging. Filter by severity.

4
Alerts

Set alerts on memory usage > 80%, hit ratio < 80%, or connected clients spike.

Key Takeaways

  • Memorystore = GCP's managed Redis. No ops, VPC-secured, auto-failover (Standard tier).
  • VPC Connector bridges Cloud Run into VPC so it can reach Memorystore's private IP.
  • Cache-Aside pattern: check cache → miss → read DB → populate cache. Evict on write.
  • Rate Limiting via INCR + EXPIRE: atomic, no race conditions, auto-resets after window.
  • Fail Open: if Memorystore is down, app degrades gracefully — just hits the database.
  • Monitor: hit ratio, latency, evictions, memory usage in Cloud Console.
