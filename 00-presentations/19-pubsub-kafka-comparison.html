Kafka vs Pub/Sub

Mental Model Shift for Kafka Administrators

Core Philosophy

Two fundamentally different worldviews on messaging

Kafka: YOU Control Consumption

Kafka is a distributed log. You decide where to read from, how fast to consume, when to commit, and when to replay. The broker stores events. You are the driver.

Model: Log + Control

Pub/Sub: SYSTEM Ensures Delivery

Pub/Sub is a delivery system. You publish, Google delivers. Messages arrive, you acknowledge. The system handles retries, scaling, and routing. Google is the driver.

Model: Delivery + Automation

This single distinction explains every difference that follows. Kafka gives you a log to navigate. Pub/Sub gives you a delivery guarantee to rely on.

No Offset Management — The Biggest Shift

The concept that trips up every Kafka administrator

Kafka: Offset-Based Consumption

Consumer reads message at offset 100.

Consumer processes message.

Consumer commits offset 101.

Consumer can reset to offset 50 and replay.

YOU control position. YOU decide what to read next.

Pub/Sub: ACK-Based Delivery

Message is delivered to subscriber.

Subscriber processes message.

Subscriber sends ACK.

Message is gone. No offset. No position.

No offset concept. No position tracking. No replay cursor.

Stop thinking in offsets. In Pub/Sub, there is no log position. There is only: delivered, acknowledged, or pending retry.

Push is completely alien to the Kafka mindset. In Push mode, the subscriber does not poll. It exposes an HTTPS endpoint, and Pub/Sub delivers messages by calling it. Think webhooks, not consumers.

ACK vs Offset Commit — Critical Difference

This is where Kafka admins get burned the most

Kafka: Offset Commit

1. Process message

2. Commit offset to __consumer_offsets

3. Offset stored in broker

4. Can reset offset and replay anytime

Message remains in the log regardless of commits.

Pub/Sub: ACK

1. Process message

2. Send ACK

3. Message removed from subscription

4. No replay by default

No offset tracking. No position management.

If you ACK, the message is gone forever from that subscription. There is no "reset offset to replay." This is the single most important behavioral difference between Kafka and Pub/Sub. Internalize it.

Retention Model

Kafka = event store. Pub/Sub = event delivery system.

Kafka Retention

Time-based: days, weeks, or forever (log compaction)

Replay is normal: consumers routinely reset offsets

Log compaction: keep latest value per key indefinitely

Kafka is designed as a durable event store.

Pub/Sub Retention

Default: 7 days (configurable up to 31 days)

Replay is limited: Seek to timestamp only

No log compaction: not an event store

Pub/Sub is designed for event delivery, not storage.

AspectKafkaPub/Sub
Primary purposeEvent log / event storeEvent delivery / routing
Retention mindset"Keep everything, replay when needed""Deliver reliably, discard after ACK"
Long-term storageBuilt-inUse BigQuery, GCS, or Dataflow

Kafka: Partition Thinking

"I need 12 partitions for 12 consumers. Messages with the same user ID go to the same partition for ordering."

Pub/Sub: Ordering Key Thinking

"I set an ordering key on the sender VPA. Pub/Sub handles parallelism and ordering internally."

Reprocessing and Replay

Kafka's strongest feature vs Pub/Sub's weakest area

Kafka Replay

Reset offset to any position: beginning, end, timestamp, specific offset

Replay is a first-class operation.

Event sourcing patterns rely on full replay.

Pub/Sub "Replay"

Seek to timestamp: limited, requires retained messages

Snapshot seek: restore subscription state to a snapshot

Re-publish: manually re-publish messages from external store

DLQ replay: reprocess failed messages from dead letter topic

Pub/Sub is NOT designed for long-term replay. If you need event sourcing or full reprocessing, store events in BigQuery or GCS at publish time and replay from there. Do not treat Pub/Sub as a Kafka-style event log.

Failure Handling

Kafka: you manage retries. Pub/Sub: retries are automatic.

Kafka: Manual Retry Control

You manage retry logic in consumer code

You control offset commits — don't commit on failure

You build DLQ by producing failed messages to a separate topic

You handle poison pills with custom error handlers

Pub/Sub: Built-in Retry

No ACK = automatic retry after ack deadline expires

NACK = immediate retry

Dead letter topic is a built-in feature with max delivery attempts

Exponential backoff configurable via retry policy

Failure ScenarioKafkaPub/Sub
Consumer crashesRebalance, another consumer picks upMessage redelivered after ack deadline
Processing failsDon't commit offset; custom retry logicNACK or let ack deadline expire
Repeated failuresManual DLQ implementationBuilt-in dead letter topic

Consumer Scaling

Kafka: manual partition management. Pub/Sub: fully automatic.

Kafka Scaling

Step 1: Estimate throughput, decide partition count up front

Step 2: Add consumers (max = partition count per group)

Step 3: Rebalance triggered — pause in consumption

Step 4: Need more parallelism? Add partitions (irreversible)

Consumer group coordination via group coordinator protocol.

Pub/Sub Scaling

Step 1: Create subscription

Step 2: Add subscribers. No limit.

That's it.

No partition management. No consumer group coordination. No rebalance storms. No ceiling on parallelism.

In Pub/Sub, scaling subscribers is like scaling web servers. Just add more instances. The system distributes messages automatically.

Infrastructure Comparison

What you manage in Kafka vs what Google manages in Pub/Sub

Kafka: You Manage

Broker cluster — 3+ nodes minimum

ZooKeeper / KRaft — metadata coordination

Disk capacity — retention drives storage needs

Network — inter-broker, client-broker connectivity

Monitoring — JMX metrics, lag tracking

Security — SSL, SASL, ACLs

Upgrades — rolling restarts, version compatibility

Pub/Sub: Google Manages

Zero clusters — no brokers to provision

Zero coordination — no ZooKeeper / KRaft

Zero disk planning — storage is automatic

Zero network config — global by default

Built-in monitoring — Cloud Monitoring integration

Built-in security — IAM, encryption at rest

Zero upgrades — always current

Pub/Sub eliminates the entire Kafka operations role. No on-call for broker failures. No disk alerts. No rebalance investigations.

UPI Transfer Event — Kafka vs Pub/Sub

Same use case, radically different operational complexity

Kafka Architecture

Topic: upi-transactions (6 partitions)

Key: sender VPA (for partition routing)

Consumer Group 1: notification-group (3 consumers)

Consumer Group 2: fraud-detection-group (3 consumers)

Consumer Group 3: analytics-group (2 consumers)

You manage: partition count, consumer group lag, offset commits, rebalancing, broker health, retention policy

Pub/Sub Architecture

Topic: upi-transaction-completed

Subscription 1: notification-sub

Subscription 2: fraud-detection-sub

Subscription 3: analytics-sub

You manage: ACK deadlines, dead letter config. That's it. Scaling, delivery, and retries are automatic.

Same business outcome. Dramatically less operational burden with Pub/Sub. The trade-off: less control over consumption behavior.

When to Use What

Architect's decision framework

Use Kafka When

Full replay capability is a hard requirement

Event sourcing is the architectural pattern

Strict ordering + fine-grained control over consumption

Log compaction for materialized views

Ultra-low latency (sub-10ms) is critical

Multi-cloud / on-prem deployment needed

Use Pub/Sub When

Decoupling services is the primary goal

Serverless scaling from zero to millions

Zero infrastructure management is desired

Event-driven microservices on GCP

Push-based delivery to Cloud Run / Functions

Global message routing across regions

On GCP, start with Pub/Sub. Move to Kafka (Confluent Cloud or self-managed) only when you need specific Kafka features that Pub/Sub cannot provide.

Key Takeaways

The shifts every Kafka administrator must internalize

  • No offsets — Stop thinking in log positions. Pub/Sub uses ACK/NACK, not offset commits.
  • ACK means gone — Once acknowledged, a message is removed. There is no "replay from offset 0."
  • Subscription = Consumer Group — Each subscription is an independent, isolated message stream.
  • No partitions to manage — Parallelism is automatic. Use ordering keys when order matters.
  • Push delivery exists — A delivery model with no Kafka equivalent. Embrace it for serverless.
  • Retries are built in — No ACK = automatic redelivery. Dead letter topics are a configuration, not custom code.
  • Zero infrastructure — No brokers, no ZooKeeper, no disks, no upgrades, no on-call.
  • Pub/Sub is a delivery system, not an event store — For long-term storage, use BigQuery or GCS.
