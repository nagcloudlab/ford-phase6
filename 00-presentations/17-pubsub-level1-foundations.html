GCP Pub/Sub — Level 1

Foundations and Mental Model

The Problem — Synchronous Call Chains

What happens when every service talks directly to every other service

Service A
→ REST →
Service B
→ REST →
Service C
Tight Coupling

Every service must know the address and API contract of every service it calls. Changes ripple everywhere.

Cascading Failures

If Service C goes down, Service B fails, which causes Service A to fail. One outage takes down the entire chain.

Slow Response Chains

Total latency = sum of all downstream calls. Users wait for the slowest service in the chain.

Hard to Scale

All services must scale together. Adding a new consumer requires redeploying the producer.

Synchronous REST chains create fragile systems that break under real-world conditions.

The Shift — Event-Driven Architecture

From direct calls to decoupled events

Producer knows every consumer. Tightly coupled. Fragile.

After: Event-Driven via Pub/Sub

Producer does not know who consumes. Consumers scale independently.

Key insight: The publisher fires and forgets. Subscribers are added or removed without touching the publisher.

Core Concept 1 — Topic

The central broadcast channel for messages

What Is a Topic?

A named resource where publishers send messages. Think of it as a broadcast channel — anyone can tune in by creating a subscription.

Mental Model

A topic is like a radio station. The station broadcasts regardless of how many listeners exist. Zero listeners? Still broadcasts. Ten listeners? All hear the same content.

naming convention
# Topic names are project-scoped resources
projects/my-project/topics/upi-transaction-completed
projects/my-project/topics/order-placed
projects/my-project/topics/user-signup
One topic per event type. Do not mix unrelated events into a single topic.

Key rule: Each subscription receives a full copy of every message. Multiple subscriptions = fan-out pattern. This is how one event reaches many consumers.

Core Concept 4 — Subscriber

The service that reads and processes messages from a subscription

Message Flow — Step by Step

What happens from the moment a message is published

1
Publish

Publisher sends a message (JSON payload + optional attributes) to a named Topic

2
Store

Pub/Sub durably stores the message and replicates it across zones for reliability

3
Deliver

Each Subscription receives its own copy of the message (pull or push delivery)

4
Process and Acknowledge

Subscriber processes the message, then sends an ACK. Pub/Sub removes the message from that subscription.

If no ACK is received within the acknowledgment deadline, Pub/Sub automatically redelivers the message. This is the at-least-once guarantee.

Key Characteristics

What makes Pub/Sub production-ready out of the box

Fully Managed

No Kafka clusters to provision. No ZooKeeper. No broker rebalancing. Google handles all infrastructure.

Highly Scalable

Handles millions of messages per second. Scales automatically with no manual partition management.

At-Least-Once Delivery

Every message is guaranteed to be delivered at least once. Messages are not lost, but may be delivered more than once.

Global Availability

Messages are replicated across zones. Multi-region by default. No single point of failure.

For Kafka users: Pub/Sub trades partition-level control for zero operational overhead. You focus on business logic, not cluster health.

Why Idempotency Matters

At-least-once delivery means your consumer may see the same message twice

Bad: No Duplicate Check
java
void handleEvent(TransactionEvent event) {
    // Process every time -- DANGEROUS
    notificationService.send(event);
    analyticsService.record(event);
    // Duplicate message = duplicate SMS,
    // duplicate analytics entry
}

User gets two SMS messages. Analytics counts the same transfer twice.

Good: Idempotent Consumer
java
void handleEvent(TransactionEvent event) {
    if (alreadyProcessed(event.getId())) {
        log.info("Duplicate, skipping");
        return; // safe exit
    }
    notificationService.send(event);
    analyticsService.record(event);
    markProcessed(event.getId());
}

Duplicates are detected and skipped. Safe, reliable processing.

Rule of thumb: If processing the same message twice causes a problem, your consumer is not idempotent. Fix it before going to production.

Pub/Sub vs Traditional Message Queue

Understanding the fundamental difference in messaging models

AspectTraditional Queue (RabbitMQ)Pub/Sub Model (GCP Pub/Sub)
ModelPoint-to-point — one consumer gets each messageFan-out — all subscriptions get every message
ConsumersCompeting consumers on a single queueIndependent subscriptions, each with its own delivery
ScalingAdd more consumers to the same queueAdd more subscriptions for new use cases
Use CaseWork distribution (process once)Event broadcast (react everywhere)
Adding ConsumersConnect to existing queueCreate a new subscription — no code change to publisher
InfrastructureSelf-managed brokersFully managed, serverless
Kafka parallel: A Pub/Sub subscription is similar to a Kafka consumer group — each subscription independently tracks its own position in the message stream.

Hands-On — Essential Commands

The four gcloud commands you will use in every Pub/Sub workflow

1. Create a Topic
bash
gcloud pubsub topics create upi-transaction-completed
2. Create a Subscription
bash
gcloud pubsub subscriptions create notification-sub \
  --topic=upi-transaction-completed \
  --ack-deadline=30
3. Publish a Message
bash
gcloud pubsub topics publish upi-transaction-completed \
  --message='{"txnId":"T001","amount":500}'
4. Pull Messages
bash
gcloud pubsub subscriptions pull notification-sub \
  --auto-ack \
  --limit=5

Common Mistakes

Misconceptions that trip up beginners

Watch Out For These

✕
Thinking topics store messages forever Pub/Sub retains unacknowledged messages for a maximum of 7 days by default. It is not a database. If no subscription exists when a message is published, that message is lost.
✕
Not handling duplicate messages At-least-once delivery means duplicates will happen. If your consumer is not idempotent, you will process the same event multiple times — leading to double charges, duplicate notifications, or corrupted analytics.
✕
Confusing subscription with consumer A subscription is a Pub/Sub resource (like a Kafka consumer group). A consumer/subscriber is your application code that reads from the subscription. Multiple consumer instances can read from the same subscription for load balancing.
✕
Publishing without any subscription If you publish to a topic with zero subscriptions, the messages are silently dropped. Always create subscriptions before publishing in production.

Key Takeaways

What you should remember from Level 1

  • Pub/Sub enables event-driven architecture — services communicate through events, not direct API calls
  • Core entities: Topic, Subscription, Subscriber — publishers send to topics, subscriptions fan out, subscribers process
  • Async and scalable communication — publishers fire and forget, subscribers process at their own pace
  • At-least-once delivery guarantee — messages are never lost, but may arrive more than once
  • Idempotent consumers are mandatory — always check for duplicates before processing
  • Fully managed and serverless — no brokers, no partitions, no ZooKeeper. Focus on business logic.
