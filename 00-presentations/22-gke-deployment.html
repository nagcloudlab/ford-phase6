Kubernetes Core Concepts

The 5 building blocks you need to know

1
Pod

Smallest deployable unit. Runs one or more containers. Gets an IP. Ephemeral — can die anytime.

2
Deployment

Declares desired state: "run 2 replicas of upi-service:latest". K8s maintains this count automatically.

3
Service

Stable network endpoint for pods. DNS name + load balancing. Pods die, Service stays.

4
ConfigMap

Externalized config (env vars, files). Change config without rebuilding images.

5
Ingress

Routes external HTTP traffic to internal Services. Single public IP for all your APIs.

6
Namespace

Logical isolation. We use upi-system to group all our services together.

Deployment YAML Explained

Anatomy of a Kubernetes Deployment

yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: upi-transfer-service
  namespace: upi-system
spec:
  replicas: 2
  template:
    spec:
      containers:
        - image: ...upi-transfer:latest
          ports:
            - containerPort: 8080
          resources:
            requests: { cpu: 250m, memory: 256Mi }
            limits:   { cpu: 500m, memory: 512Mi }
          readinessProbe:
            httpGet: /actuator/health
          livenessProbe:
            httpGet: /actuator/health
R
replicas: 2

K8s ensures 2 pods always running. One dies? New one starts automatically.

H
Health Probes

Readiness: Don't send traffic until healthy. Liveness: Restart if stuck.

R
Resources

Requests: guaranteed minimum. Limits: maximum allowed. Prevents noisy neighbors.

Deployment Steps

From code to running cluster

bash 
# 1. Create GKE cluster
gcloud container clusters create upi-cluster \
  --region=us-central1 --num-nodes=1 --machine-type=e2-medium

# 2. Create Artifact Registry repo
gcloud artifacts repositories create upi-repo \
  --repository-format=docker --location=us-central1

# 3. Build & push images (for each service)
docker build -t $REGISTRY/upi-transfer-service:latest .
docker push $REGISTRY/upi-transfer-service:latest

# 4. Apply K8s manifests
kubectl apply -f k8s/namespace.yaml
kubectl apply -f k8s/configmap.yaml
kubectl apply -f k8s/upi-transfer-service.yaml
kubectl apply -f k8s/notification-service.yaml
kubectl apply -f k8s/fraud-detection-service.yaml
kubectl apply -f k8s/analytics-service.yaml
kubectl apply -f k8s/ingress.yaml

# 5. Or just run: ./k8s/deploy.sh my-project us-central1

Scaling on GKE

Manual scaling, autoscaling, and node autoscaling

👉
Manual

kubectl scale deploy upi-transfer-service --replicas=5

Instant. You decide when and how many.

📈
HPA (Pod Autoscaler)

Scale pods based on CPU/memory.

--min=2 --max=10 --cpu-percent=70

🌏
Cluster Autoscaler

Scale nodes when pods can't be scheduled.

GKE adds/removes VMs automatically.

bash 
# Create HPA for upi-transfer-service
kubectl autoscale deployment upi-transfer-service \
  -n upi-system --min=2 --max=10 --cpu-percent=70

Rolling Updates

Zero-downtime deployments

1
Push new image

docker push .../upi-transfer-service:v2

2
Update deployment

kubectl set image deploy/upi-transfer-service upi-transfer-service=...v2

3
K8s rolls out

New pod starts → readiness check passes → old pod terminates. No downtime.

4
Rollback if needed

kubectl rollout undo deploy/upi-transfer-service

K8s only sends traffic to pods that pass the readiness probe. Users never see downtime during deploys.
Rollback is one command. K8s keeps the previous ReplicaSet. Instant undo.

GKE vs Cloud Run

When to use which?

Cloud Run

Best for: Stateless HTTP services, event-driven, low/bursty traffic

Scaling: 0 to N (scale to zero = pay nothing when idle)

Ops: Zero. No cluster, no nodes, no kubectl

Limitations: No persistent volumes, limited control, VPC connector needed

GKE

Best for: Multi-service platforms, stateful workloads, full control

Scaling: 1 to N (always-on, HPA, cluster autoscaler)

Ops: Medium. You manage deployments, but Google manages nodes

Power: Full K8s API, custom networking, service mesh, GPU workloads

Our choice: GKE — because we have 4 interconnected services, need Pub/Sub subscribers always running, and want full orchestration.

Key Takeaways

  • GKE = managed Kubernetes on GCP. Google manages the control plane, you manage workloads.
  • Deployment declares desired state (replicas, image, resources). K8s maintains it.
  • Service provides stable networking. Pods can die — Service DNS stays the same.
  • Ingress exposes HTTP services via GCP Load Balancer. One public IP for all APIs.
  • Health probes prevent traffic to unhealthy pods. Liveness restarts stuck containers.
  • Rolling updates = zero-downtime deployments. Rollback with one command.
  • deploy.sh script automates everything: cluster, images, manifests in one command.
