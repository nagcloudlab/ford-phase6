GCP Pub/Sub — Level 2

Core Mechanics for Production Systems

Pull Model Overview

The subscriber controls when and how fast messages are consumed

How Pull Works

The subscriber actively polls Pub/Sub for new messages. The consumer decides when to fetch, how many to fetch, and how fast to process.

Requires polling — no messages arrive unless the subscriber asks for them.

Mental Model: Worker Pattern

"I will take messages when I am ready."

Think of a worker at a counter — they go to the window, pick up the next task, complete it, then go back for more. The worker controls the pace.

1
Consumer-Controlled Rate

Backpressure is built in — slow consumers simply pull less often

2
Best for Backend Services

Long-running services on VMs, GKE, or Spring Boot applications

3
Requires Client Library

Subscriber needs the Pub/Sub SDK or gcloud CLI to pull messages

Push Model Overview

Pub/Sub delivers messages directly to your HTTP endpoint

How Push Works

Pub/Sub sends an HTTP POST request to a configured endpoint every time a new message arrives. No polling required — near real-time delivery.

Requires a public HTTPS endpoint that can receive POST requests.

Mental Model: Webhook Pattern

"I will send immediately when an event happens."

Think of a doorbell — the moment someone arrives, the bell rings. You do not need to keep checking the door.

1
No Polling Needed

Messages arrive at your endpoint automatically — near real-time

2
Perfect for Cloud Run

Services scale to zero and wake on demand when messages arrive

3
ACK via HTTP Status

Return 200 = acknowledged. Return 4xx/5xx = Pub/Sub retries delivery

Pull vs Push Comparison

Choose the right delivery model based on your architecture

DimensionPullPush
ControlConsumer controls the ratePub/Sub controls the rate
ScalingYou manage concurrencyAuto-scales with push config
RiskUnder-fetching (messages pile up)Overwhelming the endpoint
Use CaseBackend workers, batch processing, GKECloud Run, Cloud Functions, webhooks
Mental ModelWorker Pattern — "I fetch when ready"Webhook Pattern — "Send me immediately"
EndpointNo public endpoint neededPublic HTTPS endpoint required
Client LibraryRequiredNot required (just HTTP)
Rule of thumb: Use Pull for always-on backend services. Use Push for serverless and scale-to-zero architectures.

Pull vs Push Commands

Creating subscriptions with gcloud CLI

Create a Pull Subscription
gcloud pubsub subscriptions create notification-pull-sub \
  --topic=upi-transaction-completed \
  --ack-deadline=30
Create a Push Subscription
gcloud pubsub subscriptions create notification-push-sub \
  --topic=upi-transaction-completed \
  --push-endpoint=https://notification-service-abc123.a.run.app/pubsub/receive \
  --ack-deadline=30
Key difference: The --push-endpoint flag is what makes a subscription push instead of pull. Without it, the subscription defaults to pull.

ACK — The Most Important Concept

ACK = "I confirm I processed this message successfully"

Message Delivered to Subscriber
↓
Subscriber Processes Message
↓
ACK Sent — Message Removed from Subscription
1
ACK = Success Confirmation

Tells Pub/Sub: "I received and processed this. You can remove it."

2
NACK = Explicit Failure

Tells Pub/Sub: "I failed. Please send this message again."

3
No Response = Assumed Failure

If the subscriber crashes or times out, Pub/Sub redelivers automatically.

This is the HEART of Pub/Sub. The entire reliability model revolves around acknowledgements. Every design decision you make connects back to ACK behavior.

What Happens Without ACK?

No ACK = Pub/Sub assumes failure and sends the message again

Message Delivered
↓
Subscriber Processes (no ACK sent)
↓
ACK Deadline Expires — Pub/Sub Thinks It Failed
↓
Message Redelivered (duplicate!)
At-least-once delivery: Pub/Sub guarantees every message is delivered at least once. The same message MAY arrive multiple times. This is by design, not a bug.
Subscriber Crashes

No ACK sent. Message redelivered to another subscriber instance.

Processing Too Slow

ACK deadline passes before processing finishes. Redelivered as duplicate.

Network Failure

ACK sent but lost in transit. Pub/Sub never received it. Redelivered.

ACK Failure — Real-World UPI Scenario

When ACK fails, real money problems happen

Scenario: Double Charge on Order #123

1
Message Arrives

Order #123: Transfer Rs.5000 from user-A to merchant-B

2
Processing Succeeds

Payment processed. Rs.5000 debited from user-A, credited to merchant-B.

3
ACK Fails

Network blip — ACK never reaches Pub/Sub. Pub/Sub thinks processing failed.

4
Message Redelivered

Pub/Sub sends Order #123 again. Subscriber processes it again.

5
Double Charge!

Rs.5000 debited TWICE from user-A. Customer dispute. Compliance issue.

This is not a theoretical problem. In production payment systems, this WILL happen. The solution is idempotency — covered in the next slide.

Idempotency — Survival Skill

Because duplicate messages WILL happen. Your code must handle them.

Bad: No Duplicate Protection
void processPayment(String orderId) {
    // Charges user every time it runs
    debitAccount(orderId, amount);
    creditMerchant(orderId, amount);
    // Duplicate message = double charge!
}
Good: Idempotent Processing
void processPayment(String orderId) {
    if (alreadyProcessed(orderId)) {
        log.info("Duplicate, skipping: {}", orderId);
        return; // Safe -- no double charge
    }
    debitAccount(orderId, amount);
    creditMerchant(orderId, amount);
    markAsProcessed(orderId);
}
Store Processed IDs

Keep a set of processed message or transaction IDs. Check before processing.

Use DB Uniqueness

Add a unique constraint on transaction ID. Let the database reject duplicates.

Use Status Flags

Track processing status (PENDING, COMPLETED). Only process if PENDING.

ACK Deadline — Hidden Trap

Pub/Sub gives you a limited window to acknowledge. Miss it and duplicates appear.

Message Delivered (clock starts: 10 seconds default)
↓
Processing takes 15 seconds...
↓
Deadline expires at 10s — Pub/Sub assumes FAILURE
↓
Message redelivered — duplicate processing even though code is correct
Your code may be perfectly correct, but if it takes too long, Pub/Sub will send the message again. The deadline is the silent source of most duplicate issues.
Fix 1: Extend Deadline

Set --ack-deadline=60 or use modifyAckDeadline API to extend during processing.

Fix 2: Process Faster

Offload heavy work to background threads. ACK quickly, process asynchronously.

Fix 3: Async Processing

ACK immediately, write to a local queue or DB, process later at your own pace.

Manual ACK Commands

Pull messages without auto-acknowledgement, then ACK explicitly

Step 1: Pull Without Auto-ACK
gcloud pubsub subscriptions pull notification-sub \
  --auto-ack=false \
  --limit=1 \
  --format=json
Step 2: Process the Message (inspect data, run business logic)
# The pull response includes an ACK_ID for each message
# Example ACK_ID: "XXXXXXXXX-YYYYY-ZZZZZ"
Step 3: Acknowledge with ACK ID
gcloud pubsub subscriptions ack notification-sub \
  --ack-ids="XXXXXXXXX-YYYYY-ZZZZZ"
Manual ACK gives you full control. Message stays in the subscription until you explicitly confirm successful processing.

Retry Behavior — Automatic and Infinite

Pub/Sub retries failed messages forever unless you configure limits

Message Delivered
↓
Processing Fails (NACK or timeout)
↓
Pub/Sub Waits, Then Retries
↓
Fails Again
↓
Retries Again... and Again... Forever
Poison Message Problem: A malformed or corrupt message that always fails will be retried infinitely, wasting resources and blocking other messages. Without a Dead Letter Queue, there is no escape.
Default Behavior

Retry is automatic. There is no retry limit by default. Exponential backoff applies between retries.

The Solution

Configure a Dead Letter Queue (DLQ) to catch messages that fail repeatedly and stop the infinite loop.

Dead Letter Queue — Safety Net

Stop infinite retry loops. Capture failed messages for investigation.

Why DLQ?

Without DLQ, poison messages retry forever, wasting compute and blocking progress. DLQ breaks the cycle.

How It Works

After N failed delivery attempts, Pub/Sub automatically moves the message to a designated dead-letter topic.

What You Do With DLQ

Debug the failure. Fix the data or code. Replay corrected messages back to the original topic.

1
Message Fails Repeatedly

Subscriber NACKs or times out on the same message multiple times

2
Max Attempts Reached

After the configured max delivery attempts (e.g., 5), Pub/Sub stops retrying

3
Moved to DLQ

Message is published to the dead-letter topic for manual inspection

4
Alert and Investigate

Set up monitoring on the DLQ. Investigate, fix, and replay as needed.

DLQ Commands

Setting up a Dead Letter Queue with gcloud CLI

Step 1: Create the Dead Letter Topic
gcloud pubsub topics create upi-transaction-dead-letter
Step 2: Create a Subscription on the Dead Letter Topic (for monitoring)
gcloud pubsub subscriptions create dead-letter-monitor-sub \
  --topic=upi-transaction-dead-letter
Step 3: Create Subscription with DLQ Configured
gcloud pubsub subscriptions create notification-sub \
  --topic=upi-transaction-completed \
  --dead-letter-topic=upi-transaction-dead-letter \
  --max-delivery-attempts=5 \
  --ack-deadline=30
After 5 failed delivery attempts, the message is automatically forwarded to the dead-letter topic instead of being retried forever.

DLQ Flow Diagram

End-to-end message lifecycle with Dead Letter Queue

Publisher sends message
↓
Topic: upi-transaction-completed
↓
Subscription: notification-sub
↓
Subscriber receives and processes
↓
ACK? YES → Done. Message removed.
ACK? NO → Retry (up to max-delivery-attempts)

↓

Every production Pub/Sub subscription should have a DLQ configured. Without it, poison messages create silent infinite loops.

Message Ordering

Default: no ordering guarantee. Ordering requires explicit configuration.

Default: No Ordering

Messages may arrive in any order. Sent: A, B, C. Received: B, A, C.

Fine for independent events like notifications or analytics.

With Ordering Keys

Messages with the same ordering key are delivered in order. Different keys are processed in parallel.

Required for: Transfer → Refund sequences on the same account.

Enable Ordering on Subscription
gcloud pubsub subscriptions create ordered-sub \
  --topic=upi-transaction-completed \
  --enable-message-ordering
Publish with Ordering Key
gcloud pubsub topics publish upi-transaction-completed \
  --message='{"orderId":"123","amount":5000}' \
  --ordering-key=user-A-vpa
Warning: Ordering reduces parallelism. Messages with the same ordering key are processed sequentially. Only use ordering when sequence truly matters.

Message Attributes (Metadata)

Every message = Data (payload) + Attributes (key-value metadata)

Publish with Attributes
gcloud pubsub topics publish upi-transaction-completed \
  --message='{"orderId":"123","amount":5000,"senderVpa":"user-A@upi"}' \
  --attribute=eventType=TRANSFER,priority=HIGH,region=south
Filtering

Subscriptions can filter messages by attributes without parsing the payload. Process only what you need.

Routing Logic

Route messages to different handlers based on eventType or region without deserializing.

Event Typing

Use attributes to indicate message schema version or event type for backward compatibility.

Create Subscription with Filter
gcloud pubsub subscriptions create high-priority-sub \
  --topic=upi-transaction-completed \
  --message-filter='attributes.priority = "HIGH"'

Push Subscription Security

Anyone can hit your endpoint unless you authenticate push requests

Problem: Unauthenticated Push

Your push endpoint is public. Anyone who discovers the URL can send fake messages to your service, triggering unauthorized actions.

Solution: Authenticated Push

Configure a Service Account for push delivery. Pub/Sub includes a signed JWT token in each request. Your service verifies the token before processing.

Create Authenticated Push Subscription
gcloud pubsub subscriptions create secure-push-sub \
  --topic=upi-transaction-completed \
  --push-endpoint=https://notification-service.a.run.app/pubsub/receive \
  --push-auth-service-account=pubsub-invoker@ford-upi-project.iam.gserviceaccount.com
Required for Cloud Run: Cloud Run services reject unauthenticated requests by default. You must configure push authentication with a service account that has the roles/run.invoker role.

Common Mistakes

Pitfalls that cause production incidents in Pub/Sub systems

Mistakes That Will Hurt You in Production

1
Not Handling Duplicate Messages

At-least-once delivery means duplicates are guaranteed. Without idempotency, you get double charges, duplicate notifications, and corrupted data.

2
Ignoring ACK Deadlines

Default is 10 seconds. If processing takes longer, messages are redelivered even though your code is working correctly. Silent duplicates.

3
No Dead Letter Queue Configured

Without DLQ, a single poison message retries forever, consuming resources and potentially blocking ordered message processing.

4
Using Push Without Authentication

Unauthenticated push endpoints can be exploited. Anyone who discovers the URL can inject fake messages into your system.

5
Assuming Message Ordering

Pub/Sub does NOT guarantee order by default. If your logic depends on sequence, you must explicitly enable ordering keys.

Key Takeaways

What you must remember from Level 2

  • Pull = consumer-driven, Push = service-driven — choose based on your architecture and scaling model
  • ACK controls the entire message lifecycle — no ACK means redelivery, always
  • Redelivery is expected behavior, not a bug — at-least-once delivery is by design
  • Idempotency is mandatory — every subscriber must handle duplicate messages safely
  • ACK deadline is a silent trap — slow processing causes duplicates even when code is correct
  • Dead Letter Queue prevents infinite retry loops — configure DLQ on every production subscription
  • Message ordering requires explicit configuration — use ordering keys only when sequence matters
  • Attributes enable event-driven filtering and routing — use metadata to avoid parsing payloads
  • Authenticate push endpoints — unauthenticated push is a security risk in production
