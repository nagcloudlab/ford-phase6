BigQuery

Streaming Analytics for UPI Transactions

The Problem

Our analytics are ephemeral — they die with the pod

💣
In-Memory = Lost on Restart

analytics-service stores stats in RAM. Pod restarts? All data gone. Zero history.

🔍
No Ad-Hoc Queries

"Show me all transactions > 5000 from last week" — impossible with in-memory counters.

📊
No Dashboards

Business wants charts: daily volume, top senders, fraud trends. Can't build on volatile data.

🚀
Scale Problem

1M transactions/day — you can't store that in a Java Map. You need a data warehouse.

Solution: Stream every transaction event from Pub/Sub into BigQuery. Persistent, queryable, scalable.

What is BigQuery?

Google's serverless, petabyte-scale data warehouse

1
Serverless

No clusters, no VMs, no capacity planning. Just create a table and query. Google handles infra.

2
Standard SQL

Write familiar SQL. Joins, aggregations, window functions — all supported. No new query language.

3
Columnar Storage

Data stored by column, not row. Scanning 1 column of 1B rows is fast. Perfect for analytics.

4
Streaming Inserts

Push rows in real-time via API. Available for query within seconds. No batch ETL needed.

5
Pay Per Query

Storage: ~$0.02/GB/month. Queries: $5/TB scanned. First 1TB/month free. 10GB storage free.

6
Petabyte Scale

Query terabytes in seconds. No indexes needed — BigQuery scans columns in parallel across thousands of nodes.

Our Table Schema

ColumnTypeDescription
transaction_idINT64Unique transaction ID from UPI service
sender_upi_idSTRINGWho sent the money (nag@upi)
receiver_upi_idSTRINGWho received the money (ram@upi)
amountNUMERICTransfer amount (precise decimal)
statusSTRINGSUCCESS or FAILED
timestampTIMESTAMPWhen the transfer happened

ingested_at lets you measure pipeline latency: ingested_at - timestamp = how fast events flow from transfer to BigQuery.

Streaming Insert API

Real-time data ingestion into BigQuery

Java 
// Build a row
Map<String, Object> row = new HashMap<>();
row.put("transaction_id", event.getTransactionId());
row.put("sender_upi_id", event.getSenderUpiId());
row.put("amount", event.getAmount());
row.put("timestamp", event.getTimestamp());
row.put("ingested_at", Instant.now());

// Stream into BigQuery
InsertAllRequest request = InsertAllRequest
    .newBuilder(tableId)
    .addRow(row)
    .build();

InsertAllResponse response = bigQuery.insertAll(request);

if (response.hasErrors()) {
    // handle errors
}

Querying with SQL

The real power — ask any question about your transaction data

Daily Volume
SELECT DATE(timestamp) AS day,
       COUNT(*) AS txn_count,
       SUM(amount) AS total_volume
FROM upi_analytics.transactions
GROUP BY day ORDER BY day
Top Senders
SELECT sender_upi_id,
       COUNT(*) AS txn_count,
       SUM(amount) AS total_sent
FROM upi_analytics.transactions
WHERE status = 'SUCCESS'
GROUP BY 1 ORDER BY 3 DESC
Failed Transfers
SELECT sender_upi_id, amount,
       timestamp
FROM upi_analytics.transactions
WHERE status = 'FAILED'
ORDER BY timestamp DESC
LIMIT 10
Pipeline Latency
SELECT
  AVG(TIMESTAMP_DIFF(
    ingested_at, timestamp, SECOND
  )) AS avg_latency_sec
FROM upi_analytics.transactions

Batch vs Streaming

Two ways to get data into BigQuery

Batch Load

How: Upload CSV/JSON/Parquet files via bq load or Storage API

Speed: Minutes to hours (depends on file size)

Cost: Free! Batch loads don't cost anything

When: Historical data migration, nightly ETL, large bulk imports

Streaming Insert (our approach)

How: insertAll() API call per row/batch from application code

Speed: Seconds (queryable almost immediately)

Cost: $0.01/200MB (first 2TB/month free)

When: Real-time analytics, event streams, live dashboards

We use streaming inserts because transactions happen in real-time and we want instant queryability.

Deploying on GCP

Cloud Run + Pub/Sub + BigQuery

bash 
# 1. Enable BigQuery API
gcloud services enable bigquery.googleapis.com

# 2. Build and push image
gcloud builds submit \
  --tag gcr.io/$PROJECT_ID/analytics-bq

# 3. Deploy to Cloud Run
gcloud run deploy analytics-bq \
  --image=gcr.io/$PROJECT_ID/analytics-bq \
  --set-env-vars="GCP_PROJECT_ID=$PROJECT_ID,\
BQ_DATASET=upi_analytics,BQ_TABLE=transactions" \
  --vpc-connector=upi-connector \
  --no-cpu-throttling \
  --min-instances=1

# 4. Query your data!
bq query --use_legacy_sql=false \
  'SELECT * FROM upi_analytics.transactions
   ORDER BY timestamp DESC LIMIT 5'
--no-cpu-throttling

Keeps CPU active between requests. Required for Pub/Sub subscriber (always listening).

--min-instances=1

Always keep one instance warm. Pub/Sub needs a listener running at all times.

IAM

Cloud Run service account needs bigquery.dataEditor and pubsub.subscriber roles.

Production Considerations

1
Batch Streaming Inserts

Don't insert 1 row at a time in production. Buffer 100-500 rows and insert in one API call. Much more efficient.

2
Deduplication

Streaming is at-least-once. Use insertId (transaction_id) to let BigQuery dedup. Or use DISTINCT in queries.

3
Partitioning

Partition by timestamp (day). Queries that filter by date scan only relevant partitions = faster + cheaper.

4
Cost Control

Set project-level query quotas. Use --maximum_bytes_billed to prevent expensive queries.

5
Alternatives

For simple cases: Pub/Sub → BigQuery subscription (zero code). For complex: Dataflow (Apache Beam).

6
Access Control

Grant bigquery.dataViewer to analysts, bigquery.dataEditor to services. Dataset-level or table-level IAM.
