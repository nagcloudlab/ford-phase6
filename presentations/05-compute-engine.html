What is a VM, Really?

A virtual computer running inside Google's data center

Physical Server (Google's Hardware)
↓
Hypervisor (Virtualization Layer)
↓
Virtual Machine (Your Instance)
↓
Operating System (Linux / Windows)
↓
Your Application (Spring Boot, Node, etc.)
You get a full computer — CPU, RAM, disk, network — without owning any hardware.

VM Building Blocks

The four components you configure when creating a VM

1. Machine Type (CPU + RAM)

TypeSpecsUse Case
e2-micro0.25 vCPU, 1GBFree tier, testing
e2-standard-22 vCPU, 8GBSmall apps
e2-standard-88 vCPU, 32GBProduction
2. Boot Disk

Standard (HDD) or SSD (faster, for prod)

3. OS Image

Ubuntu, Debian, CentOS, Windows Server

4. Networking

Internal IP (always) + External IP (optional)

Hands-On: Create Your First VM

Two ways — Console UI or command line

Via Console

1
Navigate

Compute Engine → VM Instances

2
Click Create Instance

Name: my-first-vm

3
Configure

Region, Zone, Machine type, OS

4
Create

Ready in ~30 seconds

Via Command Line

bash
gcloud compute instances create my-first-vm \
  --zone=asia-south1-a \
  --machine-type=e2-micro \
  --image-family=ubuntu-2204-lts \
  --image-project=ubuntu-os-cloud
Your VM is ready in ~30 seconds.

Hands-On: SSH & Deploy

Connect, install, run, and access your application

1. Connect via SSH

Click the SSH button in Console — browser-based terminal

2. Install Java
sudo apt update
sudo apt install openjdk-17-jdk -y
3. Run Your App
java -jar app.jar
4. Open Firewall & Access

Allow TCP:8080 in VPC Firewall, then visit http://EXTERNAL-IP:8080

When to Use VMs (and When NOT To)

Choose the right tool for the job

Use VMs When
✅
Full OS control — custom kernel, specific drivers
✅
Legacy applications — can't be containerized
✅
Custom dependencies — need specific setup
✅
Learning infra — understanding fundamentals
Avoid VMs When
🔴
Building microservices → use Cloud Run
🔴
Want auto-scaling → VMs require manual scaling
🔴
Hate maintenance → VMs need OS patching

Expert Takeaways

What you should remember from this section

  • VMs = maximum control, minimum automation
  • You manage: OS, security patches, scaling, monitoring — all of it
  • Great for: Learning cloud fundamentals & running legacy systems
  • Not ideal for: Modern cloud-native microservices
  • Always remember: Stop what you're not using — your wallet will thank you
