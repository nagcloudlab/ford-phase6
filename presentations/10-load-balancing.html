TypeScopeBest For

TypeUse Case

Traffic Distribution Strategies

How the LB decides where to send each request

StrategyHow It WorksBest For
Round RobinEqual turns to each serverUniform workloads
Least ConnectionsSend to the least busy serverVariable request times
Geo RoutingRoute based on user locationGlobal applications
Geo Routing Example
India users   → Mumbai backend   (asia-south1)
US users      → Iowa backend     (us-central1)
Europe users  → Belgium backend  (europe-west1)
Users always hit the closest server. Latency stays low.

Health Checks: Keeping Your System Alive

The LB continuously checks — "Is this backend healthy?"

1
Probe

LB sends periodic requests to /health endpoint

2
Healthy

Backend responds 200 OK — keep sending traffic

3
Unhealthy

No response — remove from pool automatically

Server A — Healthy

LB → /health → 200 OK ✅ → Send traffic

Server B — Unhealthy

LB → /health → No response → Stop sending traffic

Without health checks, users get routed to dead servers.

Failover: Surviving Region Failures

Mumbai region goes completely down — what happens?

Without Failover
🔴
App is down for all users
🔴
Manual intervention needed
🔴
Could be hours of downtime
With Global LB + Multi-Region
✅
Normal: Users → Mumbai (primary)
✅
Failure: Mumbai down → LB auto-switches → Delhi
✅
Recovery: Mumbai back → LB gradually returns traffic
Automatic. No human intervention. Users may not even notice.
