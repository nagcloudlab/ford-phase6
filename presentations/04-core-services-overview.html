Compute Services: Where Your App Runs

Four options — from full control to fully managed

🖥️
Compute Engine (VMs)

Full control over OS, runtime, configs.
Best for: Legacy apps, custom setups, learning infra.

📦
App Engine (PaaS)

Deploy code, no server management.
Best for: Simple web apps, prototypes.

🚀
Cloud Run (Serverless)

Docker containers with zero infra management.
Best for: Microservices, REST APIs, modern apps.

☸️
GKE (Kubernetes)

Full Kubernetes orchestration at scale.
Best for: Large-scale microservices, complex systems.

Default modern choice: Cloud Run — unless you have a specific reason for something else.

Common Mistakes to Avoid

Patterns that seem reasonable but cause real problems

MistakeWhy It's WrongFix
Using VMs for everythingOld-school mindset, high ops burdenUse Cloud Run for microservices
Kubernetes too earlyOverengineering for small appsStart with Cloud Run, graduate to GKE
Storing files in databaseWrong tool, bad performanceUse Cloud Storage for files
Ignoring async messagingTight coupling, fragile systemUse Pub/Sub to decouple

Expert Takeaways

What you should remember from this section

  • Compute choice = the biggest architectural decision you'll make
  • Cloud Run = your default for modern apps — serverless, auto-scaling, pay-per-request
  • Storage ≠ Database — keep this separation clear in your architecture
  • Pub/Sub enables resilient, event-driven systems — decouple everything
  • Think like an architect: choose services by need, not by popularity
