Projects: The Most Important Concept

Everything in GCP lives inside a Project

🖥️
Compute Engine VMs

Virtual machines for running workloads

🚀
Cloud Run Services

Serverless containers for modern apps

🗄️
Storage Buckets

Object storage for files and media

🗃️
Databases

Cloud SQL, Firestore, Bigtable

Critical rule: If a project is deleted → EVERYTHING inside is gone. VMs, databases, storage — all of it.

Real-World Example: Project Structure

How teams should organize GCP projects

Hierarchy
Organization: acme-corp.com
   └── Folder: payments
         ├── Project: payments-prod
         ├── Project: payments-dev
         └── Project: payments-test
Golden rule: Always separate PROD from NON-PROD.
The Beginner Mistake

Everything in ONE project → access chaos, billing confusion, no isolation.

The Right Pattern

Separate by environment: myapp-dev, myapp-test, myapp-prod

IAM: Who Can Do What, Where?

Identity & Access Management — the security backbone of GCP

WHO (Identity)
+
WHAT (Role)
+
WHERE (Resource)
ComponentQuestionExample
WhoWhich user or service?dev@company.com
WhatWhich role?Viewer
WhereWhich resource?myapp-prod project
Result: dev@company.com can only VIEW resources in myapp-prod. Nothing more.

IAM Roles: From Broad to Precise

The principle of least privilege in action

Basic Roles — Too broad for production

RoleAccess Level
OwnerFull control + billing
EditorCreate & modify resources
ViewerRead-only access

Predefined Roles — Use these!

Compute Admin

Manage VMs only

Storage Admin

Manage buckets only

Cloud Run Admin

Manage Cloud Run only

Principle of Least Privilege: Give the minimum access needed. Nothing more.

The Classic IAM Disaster

What happens when access is not managed properly

Bad: Owner Role to All 20 Devs
🔴

🔴

🔴

Good: Specific Roles Per Person

✅
DBA → Cloud SQL Admin
✅
Intern → Viewer only

Your app needs to access Cloud Storage. How does it authenticate?

Wrong Way

Billing: Where Money Meets Architecture

Billing is linked to PROJECT — no billing account means most services won't work

Real Risks

ScenarioConsequence
No budget alertsSurprise bill of ₹50,000
Forgot to stop a VMCharges running 24/7
No resource cleanupZombie resources eating money

Best Practices

Set Budget Alerts

At 50%, 80%, and 100% thresholds

Clean Up Daily

Stop/delete unused resources

Use Free Tier

Use wisely during learning

Expert Takeaways

What you should remember from this section

  • Project = isolation boundary + billing boundary — the core unit of GCP
  • IAM = the security backbone of all GCP — Who + What + Where
  • Service Accounts = mandatory for every app — never hardcode credentials
  • Least Privilege = the #1 security principle — minimum access needed
  • Billing control = an architecture responsibility, not an afterthought
